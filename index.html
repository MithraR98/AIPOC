Best Prompting Practices

Prompting is becoming a core design skill — not a tool trick. The quality of prompts directly impacts layout relevance, workflow accuracy, controllability, iteration speed, and even credit consumption.

🧱
Think in Layers, Not Single Prompts

Strong UI outputs rarely come from a single instruction block. Instead of generating the full experience at once, build up progressively:

  1. Define layout structure
  2. Define content hierarchy
  3. Add workflow constraints
  4. Refine interaction behavior
  5. Apply visual system guidance
Why it matters: Layered prompting improves precision, workflow accuracy, refinement control, and output consistency.
🎯
Define User Goals Clearly
✅ DO

Describe user intent and workflow outcomes.

"Create a provider search workflow optimized for operations teams reviewing high-volume healthcare records."
🚫 DON'T

Describe only screens or visuals.

"Create a modern provider search screen."
Why it matters: AI performs better when it understands who the user is, what they are trying to accomplish, and why the workflow exists.
✂️
Prompt Smaller Workflow Segments
✅ DO

Break workflows into smaller promptable units:

  • Search flow
  • Results table
  • Filters
  • Empty states
  • Detail panels
🚫 DON'T

Generate entire enterprise workflows in a single prompt. Large prompts often:

  • Reduce controllability
  • Increase hallucinations
  • Create inconsistent workflows
Why it matters: Smaller prompt scopes improve refinement precision and iteration quality.
🏗️
Prioritize Structure Before Visual Polish
✅ DO

Start with:

  • Layout
  • Hierarchy
  • Workflow structure
  • Information architecture

Validate the foundation first.

🚫 DON'T

Spend credits refining styling before validating structure.

Why it matters: Weak structure hidden behind polished visuals creates expensive rework later.
⚙️
Add Constraints Early
✅ DO

Include:

  • Business rules
  • Accessibility expectations
  • Workflow dependencies
  • Operational limitations
  • Data density expectations
🚫 DON'T

Assume AI understands domain logic automatically. AI may generate:

  • Impossible actions
  • Incorrect states
  • Missing dependencies
  • Oversimplified workflows
Why it matters: Enterprise and healthcare workflows require explicit guidance.
♻️
Reuse Stable Prompt Patterns
✅ DO

Build reusable prompt templates for:

  • Tables
  • Dashboards
  • Form flows
  • Search experiences
  • Workflow-heavy layouts
🚫 DON'T

Rewrite prompts from scratch repeatedly.

Why it matters: Reusable prompting patterns improve consistency, speed, controllability, and credit efficiency.
🔄
Avoid Endless Regeneration
✅ DO

Diagnose what failed before iterating:

  • 1. Identify the issue
  • 2. Refine the prompt intentionally
  • 3. Add better constraints
  • 4. Regenerate strategically
🚫 DON'T

Repeatedly click regenerate hoping the output improves randomly.

Why it matters: Uncontrolled iteration is one of the biggest sources of unnecessary credit consumption.
🔭
Use AI for Exploration, Not Final Decision-Making
✅ DO — Use AI for:
  • Layout exploration
  • Early-stage ideation
  • Workflow scaffolding
  • Divergent thinking
🚫 DON'T — Rely on AI for:
  • Nuanced UX judgment
  • Accessibility validation
  • Business-critical decisions
  • Domain-heavy workflow logic
Why it matters: AI accelerates execution, but still requires human evaluation and validation.
🔬
Anatomy of an Effective UI Prompt

A strong UI prompt usually includes these layers:

LayerPurpose
User ContextWho is using the experience
Workflow GoalWhat task they need to accomplish
ConstraintsBusiness rules, accessibility, compliance
Layout GuidanceHierarchy, tables, navigation, panels
Interaction ExpectationsFilters, validation, bulk actions
Visual DirectionStyling guidance after structure is stable
⚡
Reality Check
✅ AI can accelerate:
  • Layout generation
  • Workflow scaffolding
  • UI exploration
  • Early-stage iteration
🚫 But still struggles with:
  • Domain-heavy workflows
  • Accessibility validation
  • Nuanced UX decisions
  • Business logic accuracy
  • Product prioritization
Key Takeaway: AI assists execution. It does not replace product thinking.

Claude.md

Your project's instruction manual for Claude. Simple, effective, essential.

What is Claude.md?

Claude.md is a simple text file that lives in your project folder. When you start working with Claude, it automatically reads this file first. Think of it like giving Claude a quick briefing document about your project before every conversation—your design system colors, your preferred naming conventions, your "never do this" rules. It's Claude's cheat sheet for working on YOUR specific project.

Why is it needed?

Without Claude.md, you'd start every chat from scratch. You'd repeatedly explain "we use this color palette," "buttons should always have rounded corners," "never use inline styles," and so on. It's exhausting and wastes time.

With Claude.md, Claude already knows your project rules. You write them once, and Claude remembers them forever. Every component it generates, every code suggestion it makes—all automatically aligned with your project standards. It's like having a teammate who actually reads and remembers the project documentation.

When to use?

Use Claude.md when:

  • You have a design system — Colors, typography, spacing rules that need to be followed consistently
  • You're working on a real project — Not just experimenting, but building something that will ship
  • You have naming conventions — Specific ways you name CSS classes, components, or files
  • Multiple people are involved — Your whole team can use Claude and get consistent results
  • You've learned lessons the hard way — Browser bugs, accessibility gotchas, things that broke before and shouldn't happen again

How to write Claude.md effectively

A good Claude.md file for designers includes these sections with real examples:

🎨Design Tokens & Spacing
+

What to include: Your colors, typography, and spacing values. Write them in CSS variable format so Claude can copy-paste directly into code.

## Design TokensColors: --primary: #5B5BD6; --primary-light: #E8E8F8; --text-primary: #1A1D29; --text-secondary: #4A5568; --surface: #FFFFFF; --border: #E5E7EB;Spacing (8px base grid): xs: 4px sm: 8px md: 16px lg: 24px xl: 32px 2xl: 48pxTypography: H1: 32px / 1.2 · 700 weight H2: 24px / 1.3 · 600 weight Body: 16px / 1.6 · 400 weight Small: 14px / 1.5 · 400 weight

💡 Pro tip: Use exact CSS variable syntax so Claude can copy-paste into your stylesheets.

🧩Component Patterns
+

What to include: Your naming conventions and component structure. Show actual class names and patterns so Claude follows your exact style.

## ComponentsNaming: BEM methodology Block: .card Element: .card__header, .card__body, .card__footer Modifier: .card--featured, .card--largeButton patterns: Base: .btn Primary: .btn .btn--primary Secondary: .btn .btn--secondary Sizes: .btn--sm, .btn--md, .btn--lgLayout containers: .container (max-width: 1200px) .container--narrow (max-width: 800px)NEVER use: - Generic names (button1, box2) - camelCase for classes - Inline styles (use utility classes)

💡 Pro tip: Show actual examples of class names you use, not just rules. Claude learns better from examples.

⚙️Tech Stack
+

What to include: What frameworks and libraries you're using, and importantly, what you're NOT using. Include the "why" when it matters.

## StackHTML: Vanilla — no framework (React/Vue forbidden)CSS: Custom properties + utility classes — no TailwindJS: Vanilla ES6 modules — no jQuery, no bundlerIcons: Tabler Icons (CDN) — ti-* class namesFonts: Roboto (body) + Roboto Mono (code/data)> Why vanilla? Client maintains the site post-handoff> and doesn't have frontend devs on staff.

💡 Pro tip: Include the "why" for tech choices so Claude understands your constraints, not just preferences.

🚫Constraints & Gotchas
+

What to include: Your "never do this" rules, browser constraints, accessibility requirements, and past bugs you've fixed.

## Never Do This❌ NEVER use jQuery — client requirement❌ NEVER use position: fixed for modals — breaks iOS Safari❌ NEVER forget alt text on images — accessibility requirement❌ NEVER use px for font sizes — use rem for accessibility❌ NEVER mix inline styles — always use classes## Browser SupportMust support: Chrome, Firefox, Safari (desktop + mobile)IE11 support: NO (confirmed with client)## Past Bugs FixedBUG-001: Safari flex gap issue Problem: gap property not supported in Safari 14.0 Solution: Use margin on child elements insteadBUG-002: Z-index conflicts with header Problem: Modals appearing behind sticky header Solution: Header uses z-index: 1000, modals use 2000+

💡 Pro tip: Number your bugs (BUG-001, BUG-002) so you can reference them in future chats with Claude.

📁File Structure
+

What to include: Your folder organization with exact paths and naming rules. Show the tree structure so Claude knows where to create files.

## File Structuresrc/ components/ button.css card.css modal.css navigation.css layouts/ header.css footer.css sidebar.css pages/ dashboard.html profile.html settings.html utils/ helpers.js validators.js main.css main.jsRULES:- One component per file- CSS files match component names exactly- NEVER create new top-level folders without approval- HTML pages go in pages/, not root

💡 Pro tip: Show the actual tree structure so Claude knows exactly where to put new files you ask it to create.

✨Code Style Guidelines
+

What to include: Specific formatting rules for HTML, CSS, and JavaScript so generated code matches your style perfectly.

## Code StyleHTML:- Indent: 2 spaces (not tabs)- Attributes: double quotes- Self-closing: no trailing slash (<img> not <img />)- Boolean attributes: no value (disabled, not disabled="true")- Order: id, class, data-*, other attributesCSS:- Indent: 2 spaces- Property order: position, display, layout, typography, colors- Space after colon, none before- Always end with semicolon (even last property)- One selector per line for multiple selectorsJavaScript:- Indent: 2 spaces- Use const/let, NEVER var- Single quotes for strings- Semicolons required- Function declarations: function name() {}- Descriptive variable names (not x, y, z)

💡 Pro tip: Be hyper-specific about formatting. The more detail you give, the more consistent Claude's code will be.

💡 Overall Pro tip
Be specific, not general. Don't write "use good spacing"—write "use 8px, 16px, 24px, 32px spacing values only." Don't write "follow best practices"—write "buttons must have at least 44px touch target for mobile." The more specific you are, the more accurate Claude's suggestions will be.
🎨
How it differs from Design System?

Your Figma design system and Claude.md work together but serve different purposes. Figma shows WHAT things should look like. Claude.md tells Claude HOW to code them with your specific tech choices and conventions.

AspectClaude.mdDesign System (Figma)
PurposeHOW to code it—tech stack, class names, code rulesWHAT it looks like—visual appearance, colors, spacing
FormatText file with code examples and constraintsVisual mockups with component states
ContainsNaming conventions, file structure, tech choicesColor palettes, typography, spacing systems
Example"Use .btn-primary class" · "Never use jQuery"Button with hover states · Primary color #5B5BD6
Think of it this way: Your Figma design system is the "what"—it shows the finished button should be blue with rounded corners. Claude.md is the "how"—it tells Claude to use .btn-primary class, apply border-radius: 8px, and never use inline styles. You need both.
📚
How it differs from Skills?

Skills are Claude's built-in knowledge about general best practices (like "how to make good PDFs" or "React patterns"). Claude.md is YOUR specific project rules. Skills teach Claude universal principles; Claude.md teaches Claude YOUR way of doing things.

AspectClaude.mdSkills
What is itYour project's instruction manualClaude's built-in best practices
Created byYou write it for your specific projectAnthropic creates it based on industry standards
ScopeOnly this projectAll your projects everywhere
Example"Use .btn-primary" · "No jQuery" · "IE11 required""React best practices" · "How to make PDFs"
Think of it this way: Skills teach Claude "this is how good developers generally structure React apps." Claude.md teaches Claude "but on THIS project, we use our own custom pattern because our client has specific needs." Skills are universal knowledge; Claude.md is your specific implementation.

Claude Skills

Everything you need to know about Claude Skills

A way to package repeatable design expertise into reusable instructions — defined once, applied consistently every time.

What are Claude Skills?

📥
Loads a predefined instruction set
Like a mini system prompt — your rules, framework, and format are already in place.
→
🔁
Applies it consistently every time
No variation between sessions. Same structure, same standard, same quality bar.
→
📤
Produces predictable output
Every output follows the format you defined — ready to use, not ready to fix.

Where Skills Work vs Fall Short

✅Where it works
  • Repeatable tasks — UX audits, accessibility checks
  • Structured transformation — SOW → Persona, flows → journey map
  • Documentation-heavy work — design system docs, handoff specs
  • Checklist-driven processes — QA reviews, heuristic evaluations
🚫Where it fails
  • Ambiguous problems — product strategy, vision definition
  • Insight-heavy work requiring judgment, not just formatting
  • Creative exploration — layout concepts, visual direction
  • Context-heavy decisions — tradeoffs, business vs user conflicts

Quick Decision Test

Use a Skill if…
✓ Task is repeatable
✓ A framework already exists
✓ Output can be structured
✓ The task is time-consuming
Avoid if…
✗ Needs deep judgment
✗ Problem is unclear or open-ended
✗ Creativity is the goal

How to Write a Skill — Step by Step

01
Write Skill Instructions
Works best when you want clarity, repeatability, and consistent outputs

Use the 6-part anatomy above. Structure each section with ## headers — Claude uses these to parse your instructions clearly.

skill-instructions.md
## Purpose
Evaluate a UI against Jakob Nielsen's 10 usability heuristics…
## Input
Accepts UI screens (images), screen descriptions, or user flows…
## Framework / Method
Use Jakob Nielsen's 10 usability heuristics as the evaluation framework…
## Output Format
For each heuristic, list: Issue · Evidence · Impact · Severity (0–4) · Recommendation
## Rules / Constraints
Do not invent missing context. Base all findings strictly on provided input…
## Evaluation Criteria
Severity: 0 = No issue · 1 = Cosmetic · 2 = Minor · 3 = Major · 4 = Critical

Create the Skill in Claude Code

03
Invoke the Skill
Type the skill name to apply it instantly
